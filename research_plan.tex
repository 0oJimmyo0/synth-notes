\section*{Executive summary}
%=============================================================================

\textbf{Clinical problem.}
Restricted access to large-scale discharge summaries limits reproducible clinical NLP, fair benchmarking, and transparent model development across institutions and patient subgroups.

\textbf{Proposed contribution.}
We are building a \textbf{clinically grounded, leakage-aware validation framework} for \textbf{source-conditioned synthetic discharge-summary generation} with \textbf{audited embedding-space enrichment} and \textbf{privacy-risk evaluation}. The near-term goal is not to claim a new general-purpose clinical LLM, but to define how synthetic notes should be generated, audited, and reported before they are used for clinical NLP development.

\textbf{Central claim (narrow).}
\begin{quote}
An embedding-conditioned language model (ELM) can decode discharge summaries from held-out real note embeddings with measurable semantic faithfulness; if validated, low-magnitude \textbf{audited, metadata-linked} Concept Activation Vector (CAV) steering may improve representational coverage of the empirical real-note embedding manifold beyond multi-seed decoding and norm-matched random perturbation, while remaining \textbf{privacy-risk-evaluated} rather than assumed safe.
\end{quote}

\textbf{Phase 1 milestone (first manuscript-ready block).}
Vanilla ELM generation, a two-layer provenance system (whole-cohort source manifest + filtered-aligned generation manifest), leakage audit reporting, baseline faithfulness/quality audit, and initial internal coverage mapping.

\textbf{Extensions (Phase 2/3, not required for Phase 1).}
CAV axis bank and controlled steering; optional LLM editor; reference-guided external coverage analysis if available; downstream clinical entity extraction validation.

\section{Why this belongs in digital medicine}
%=============================================================================

This project is relevant to \emph{npj Digital Medicine} because it addresses how health systems and researchers can develop clinical NLP under real data-governance constraints. The contribution is not only a generator, but a \textbf{validation and governance protocol} covering:

\begin{itemize}[nosep]
  \item \textbf{Privacy-aware clinical NLP development:} synthetic text as auditable training/evaluation material, with explicit memorization and privacy-risk audits,
  \item \textbf{Model validation and transparency:} row-level provenance linking each synthetic note to a source embedding, split, and generation condition,
  \item \textbf{Representational coverage:} whether synthetic corpora occupy under-covered regions of the real-note embedding manifold, including subgroup-stratified gaps,
  \item \textbf{Clinical relevance:} whether generated discharge summaries preserve clinically meaningful content well enough to support downstream NLP tasks.
\end{itemize}

We will avoid broad claims such as ``privacy-safe clinical notes'' or ``covering the full semantic space.'' The intended wording is \textbf{privacy-audited}, \textbf{privacy-risk-evaluated}, and \textbf{empirical-manifold coverage}.

\section{Clinical contribution}
%=============================================================================

\subsection{Problem in clinical NLP}

Discharge summaries are high-value clinical text: they encode diagnoses, medications, procedures, disposition, and care trajectories. Yet access is restricted by IRB agreements, site-level governance, and privacy risk. This limits:
\begin{itemize}[nosep]
  \item reproducible benchmarking across institutions,
  \item development of models for underrepresented services and patient subgroups,
  \item transparent evaluation of whether synthetic text is fit for clinical NLP use.
\end{itemize}

\subsection{What this project offers clinically}

We study whether \textbf{source-conditioned synthetic discharge summaries} can support clinical NLP \textbf{development and evaluation} while preserving auditable links to source embeddings and metadata. Each synthetic note is tied to a real held-out source context, enabling:
\begin{itemize}[nosep]
  \item faithfulness checks against the source note neighborhood,
  \item leakage-aware reporting rather than optimistic generalization claims,
  \item subgroup-stratified coverage analysis for representation gaps.
\end{itemize}

\subsection{Concrete downstream validation endpoint (later phase)}

Phase 1 focuses on generation and validation infrastructure. A planned Phase 3 endpoint is \textbf{clinical entity extraction (NER)} on medication and diagnosis mentions in discharge summaries, comparing:
\begin{itemize}[nosep]
  \item models trained/evaluated on real notes only,
  \item models augmented with privacy-audited synthetic notes,
  \item stratified performance by note source condition (vanilla ELM vs.\ steered variants).
\end{itemize}

Alternative later endpoints include medication extraction, ICD/phenotype mention recovery, discharge-disposition labeling, or readmission-related text phenotyping. These are \textbf{extensions}, not dependencies for the Phase 1 milestone.

\section{Leakage and split integrity}
%=============================================================================

\noindent\fbox{%
\begin{minipage}{0.97\textwidth}
\vspace{0.25em}
\textbf{Leakage statement.}
The current cohort uses a \textbf{note-level and HADM-disjoint} 80/10/10 split, but it is \textbf{not fully patient-disjoint}. Notes and admissions do not overlap across train and held-out splits in the pre-filter split assignment, but many held-out patients also appear in training.

\vspace{0.4em}
\textbf{Pre-filter audited overlap (full test split, $n=33{,}180$ notes):}
\begin{itemize}[nosep,leftmargin=*]
  \item \texttt{note\_id} overlap with train: \textbf{0}
  \item \texttt{hadm\_id} overlap with train: \textbf{0}
  \item \texttt{subject\_id} overlap with train: \textbf{17,892 / 27,500} unique held-out patients (65.1\%)
  \item Patient-disjoint test rows: \textbf{9,949 / 33,180} (\textbf{30.0\%})
\end{itemize}

Dev split shows the same pattern: 0 note/HADM overlap; 9,883 / 33,179 patient-disjoint rows (29.8\%).

\vspace{0.4em}
\textbf{Important implementation correction.}
Generation and training use \texttt{encoded\_\*\_filtered} datasets created \emph{after} long-sequence filtering. Those filtered datasets are smaller than the pre-filter split:
\begin{itemize}[nosep,leftmargin=*]
  \item train: 265,434 $\rightarrow$ 262,895
  \item dev: 33,179 $\rightarrow$ 32,847
  \item test: 33,180 $\rightarrow$ 32,843
\end{itemize}
Therefore, the \textbf{canonical downstream leakage manifest must be regenerated against the filtered HF datasets}, not only the pre-filter metadata split. The pre-filter overlap table remains useful as source provenance, but official generation/evaluation joins must use the filtered-aligned manifest.

\vspace{0.4em}
\textbf{Reporting policy.}
We will report full held-out note-level results as secondary and treat \textbf{patient-disjoint held-out rows as the primary sensitivity subset} wherever sample size permits. All generation manifest rows will carry \texttt{patient\_overlap\_with\_train} and \texttt{hadm\_overlap\_with\_train} flags from the \textbf{filtered-aligned} train split.
\vspace{0.25em}
\end{minipage}}

\section{Methods overview}
%=============================================================================

\subsection{Pipeline}

\begin{center}
\small
\textbf{Real discharge note}
$\rightarrow$ \textbf{sentence embedding} (1024-d)
$\rightarrow$ \textbf{ELM decode}
$\rightarrow$ \textbf{synthetic discharge summary}
$\rightarrow$ \textbf{validation}
\end{center}

\noindent\textbf{Phase 2 extension:} audited CAV shift before ELM decode.
\textbf{Phase 3 extension:} optional constrained LLM editor \emph{only if} vanilla faithfulness is established.

\subsection{ELM (brief)}

An \textbf{embedding-conditioned language model} decodes text from a fixed source embedding rather than a free-form prompt. In our implementation, a BGE-large sentence embedding is mapped through a small adapter into a Llama-3.1-8B decoder at a reserved token position. This makes each synthetic note \textbf{source-conditioned} and auditable.

\subsection{Current cohort status}

\begin{itemize}[nosep]
  \item MIMIC-IV discharge summaries, note/HADM aligned: $\sim$332k notes embedded
  \item Full split before long-sequence filtering: train 265,434; dev 33,179; test 33,180
  \item Filtered ELM cohort used by the model: train 262,895; dev 32,847; test 32,843
  \item Rows removed by long-sequence filtering: 2,539 train; 332 dev; 337 test (3,208 total)
  \item ELM training complete: \texttt{checkpoint-8215} (Apr 2026)
  \item Pre-filter leakage audit complete; filtered-aligned leakage manifest refresh required before official vanilla rerun
\end{itemize}

\subsection{Manifest architecture}

Phase 1 now requires \textbf{two linked manifest layers}:
\begin{enumerate}[nosep]
  \item \textbf{Whole-cohort source manifest:} every embedding row, split assignment, and source metadata, preserved as the stable provenance table for the full cohort.
  \item \textbf{Filtered-aligned split manifest:} only rows that survive long-sequence filtering, with \texttt{dataset\_row\_id} matching \texttt{encoded\_\*\_filtered}. This is the canonical join target for vanilla generation and auditing.
\end{enumerate}

Later generation conditions (multi-seed vanilla, random shift, CAV, optional editor) should \textbf{not} overwrite the source manifest. Instead, they should write new generation manifests referencing the same stable \texttt{source\_row\_id}/\texttt{embedding\_row\_id} plus condition-specific fields such as axis id, $\alpha$, normalization flag, or editor model.

\section{Research questions}
%=============================================================================

\paragraph{RQ1 (Phase 1). Faithfulness and quality.}
Does vanilla, source-conditioned ELM generation produce readable discharge summaries that preserve source-note semantics on held-out embeddings?

\paragraph{RQ2 (Phase 1). Coverage baseline.}
What regions of the real-note embedding manifold are under-covered by vanilla synthetic generation, including subgroup-stratified gaps?

\paragraph{RQ3 (Phase 2). Audited enrichment.}
Does metadata-linked CAV steering improve coverage and structured diversity beyond multi-seed vanilla decoding and norm-matched random perturbation, without unacceptable faithfulness loss or increased privacy risk?

\paragraph{RQ4 (Phase 3). Clinical utility.}
Do privacy-audited synthetic notes improve downstream clinical entity extraction relative to real-note-only development?

\section{Evaluation framework}
%=============================================================================

All synthetic outputs will be evaluated under five reporting domains. Phase 1 implements faithfulness, quality, privacy/memorization, leakage stratification, and an initial coverage map.

\subsection{Faithfulness}

\begin{itemize}[nosep]
  \item source$\leftrightarrow$synthetic embedding cosine similarity,
  \item source-note retrieval rank and top-$k$ neighborhood recovery,
  \item diagnosis/medication/procedure/symptom entity overlap,
  \item section-presence similarity and contradiction/hallucination audit.
\end{itemize}

\subsection{Utility (Phase 3)}

\begin{itemize}[nosep]
  \item downstream clinical entity extraction performance,
  \item real-only vs.\ real+synthetic augmentation experiments,
  \item stratification by generation condition and patient-disjoint subset.
\end{itemize}

\subsection{Coverage}

\begin{itemize}[nosep]
  \item cluster occupancy over real-note embedding clusters,
  \item low-density region coverage,
  \item subgroup coverage by age, sex/gender, race/ethnicity, insurance, admission type, service, LOS,
  \item $k$NN precision/recall, MMD/energy distance, nearest-real-note distance.
\end{itemize}

\subsection{Diversity}

\begin{itemize}[nosep]
  \item within-source pairwise embedding distance across generation variants,
  \item corpus-level dispersion,
  \item entity, section-structure, and length variability.
\end{itemize}

\subsection{Privacy / memorization}

\begin{itemize}[nosep]
  \item exact duplicate detection against training notes,
  \item high $n$-gram overlap with training notes,
  \item nearest-neighbor lexical and embedding similarity to training notes,
  \item PHI-like pattern scans,
  \item membership-inference-style checks if feasible.
\end{itemize}

\subsection{Leakage}

\begin{itemize}[nosep]
  \item note/HADM/subject overlap audit (completed),
  \item patient-disjoint sensitivity analysis on all primary metrics,
  \item reporting stratified by overlap flags in the generation manifest.
\end{itemize}

\section{Phase 2/3 extensions}
%=============================================================================

\subsection{Phase 2: Coverage mapping and CAV enrichment}

\begin{itemize}[nosep]
  \item Build \texttt{factors.csv} and fit/audit axis bank on interpretable metadata axes (age, sex/gender, race/ethnicity, insurance, admission type, service, LOS bin, ICU flag),
  \item \textbf{Exclude disease-label axes} in the first version,
  \item Generate CAV-steered, multi-seed vanilla, and norm-matched random-shift controls,
  \item Compare faithfulness--coverage tradeoffs; select safe $\alpha$ values.
\end{itemize}

\textbf{CAV is not used as arbitrary noise.} Steering is accepted only if it improves pre-specified under-covered regions while passing faithfulness and privacy audits, and outperforms norm-matched random perturbation on interpretability and/or structured coverage.

\subsection{Phase 2/3 optional: reference-guided coverage}

If an external discharge-summary cohort becomes available (e.g., Yale notes encoded with the same BGE model), we will use it for \textbf{reference-guided coverage analysis} only: identifying clinically plausible embedding regions underrepresented in MIMIC. This informs enrichment targets; it does not license copying external PHI.

\subsection{Phase 3 optional: LLM editor}

A constrained LLM editor may improve local clinical coherence (temporal logic, section consistency) \textbf{only after} vanilla ELM passes faithfulness thresholds. The editor is not central to the core claim because it may weaken the clean ELM/CAV attribution. Any edited note must pass:
\begin{itemize}[nosep]
  \item post-edit cosine check vs.\ source embedding,
  \item entity preservation check,
  \item privacy/memorization re-audit.
\end{itemize}

\subsection{Phase 3: Downstream clinical utility}

Primary planned endpoint: \textbf{clinical entity extraction} on discharge summaries. Secondary options: medication extraction, ICD/phenotype mention recovery, disposition labeling, or readmission-related text phenotyping.

\section{Main risks and mitigation}
%=============================================================================

\begin{longtable}{p{4.2cm} p{5.2cm} p{5.0cm}}
\toprule
\textbf{Risk} & \textbf{Why it matters} & \textbf{Mitigation} \\
\midrule
Vanilla ELM not faithful enough &
  Invalidates all enrichment claims &
  Phase 1 faithfulness/quality gate before CAV or editor \\
CAV adds no value beyond sampling &
  Apparent diversity may be decoder noise &
  Multi-seed vanilla baseline \\
Random shifts match CAV coverage &
  Steering may not be interpretable enrichment &
  Norm-matched random perturbation control \\
Patient overlap inflates results &
  Held-out notes may be too easy for memorization-adjacent patients &
  Patient-disjoint primary sensitivity subset defined from the filtered-aligned manifest \\
Manifest drift after filtering &
  Pre-filter split metadata can stop matching the actual HF datasets used by the model &
  Maintain both source manifest and filtered-aligned manifest; use filtered manifest for all generation/audit joins \\
Synthetic notes memorize training text &
  Privacy and scientific invalidity &
  Duplicate, $n$-gram, NN, and PHI-pattern audits \\
Editor rewrites too much &
  Breaks source conditioning and confounds ELM/CAV claims &
  Post-edit cosine/entity checks; keep editor optional and late \\
Coverage gains are off-manifold &
  Synthetic text may not be clinically meaningful &
  Restrict enrichment to anchored real embeddings; audit entity/clinical content \\
\bottomrule
\end{longtable}

\section{Appendix: implementation notes}
%=============================================================================

\small
This appendix preserves technical detail for the engineering team without front-loading the clinical narrative.

\subsection{Generation manifest fields}

Each row should record: \texttt{source\_row\_id}, \texttt{note\_id}, \texttt{subject\_id}, \texttt{hadm\_id}, split, \texttt{generation\_condition}, source embedding id, seed, axis id/label, $\alpha$, normalization flag, decoding parameters, generated text, word count, success/empty/collapse flags, \texttt{patient\_overlap\_with\_train}, \texttt{hadm\_overlap\_with\_train}.

The stable source manifest should additionally preserve \texttt{embedding\_row\_id}, full-split \texttt{dataset\_row\_id\_full}, filtered \texttt{dataset\_row\_id} if retained, and a \texttt{kept\_after\_filter} flag.

Recommended \texttt{generation\_condition} values for later phases: \texttt{vanilla}, \texttt{multiseed\_vanilla}, \texttt{random\_shift}, \texttt{cav\_steered}, \texttt{high\_alpha\_cav}, \texttt{elm\_edited}.

\subsection{ELM implementation}

\begin{itemize}[nosep]
  \item Encoder: \texttt{BAAI/bge-large-en-v1.5} (1024-d)
  \item Decoder: Llama-3.1-8B with adapter injection (\texttt{open-elm/src/model.py})
  \item Standard generation input: prepared HF dataset \texttt{encoded\_testing\_filtered}, not raw \texttt{.npy} alone
\end{itemize}

\subsection{CAV axis-bank workflow (Phase 2)}

\begin{enumerate}[nosep]
  \item prepare \texttt{factors.csv},
  \item fit axis bank (\texttt{cav\_axis/fit\_axis\_bank.py}),
  \item audit axes (\texttt{cav\_axis/audit\_axis\_bank.py}),
  \item select 1--3 trusted metadata axes,
  \item steer with $\tilde{\mathbf{e}} = \mathrm{normalize}(\mathbf{e} + \alpha \mathbf{u}_j)$,
  \item initial grid: one axis at a time; $\alpha \in \{-0.5,-0.25,0.25,0.5\}$; normalize after steering.
\end{enumerate}

\section{Conclusion}
%=============================================================================

This project should be read as a \textbf{clinically grounded, leakage-aware validation framework} for source-conditioned synthetic discharge-summary generation. Phase 1 asks a disciplined question: can we generate and audit synthetic notes from held-out real embeddings with acceptable faithfulness, quality, and privacy-risk profile? Only if that gate is passed do we test whether audited CAV steering improves representational coverage beyond proper baselines, and later whether the resulting corpus has clinical NLP utility.
